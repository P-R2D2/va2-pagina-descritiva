Arquitetura

A aplicação foi dividida em três serviços independentes, orquestrados pelo Dokploy sobre Docker Swarm em um único nó (VM Ubuntu local). Cada serviço é construído a partir de um repositório no GitHub, com build automático via Dockerfile e deploy disparado por push.

02

Infraestrutura base

Provisionamento da VMconcluído

VM Ubuntu provisionada localmente no Oracle VirtualBox, com 40GB de disco (redimensionado de 20GB após esgotamento de espaço — ver seção de incidentes) e adaptador de rede em modo Bridge, permitindo acesso direto pela rede local.

Acesso e segurançaconcluído

Acesso remoto via SSH habilitado, firewall configurado com UFW liberando as portas necessárias.

# Atualização do sistema
sudo apt update && sudo apt upgrade -y

# SSH
sudo apt install openssh-server -y
sudo systemctl enable ssh

# Firewall
sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw allow 3000/tcp
sudo ufw allow 8080/tcp
sudo ufw allow 8081/tcp
sudo ufw enable
Dockerconcluído
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh
sudo usermod -aG docker $USER
03

Dokploy

Instalado via script oficial, que provisiona Docker Swarm, Traefik (proxy reverso), Postgres e Redis internos automaticamente.

curl -sSL https://dokploy.com/install.sh | sh

Painel administrativo acessível em http://<ip-da-vm>:3000, com usuário administrador configurado no primeiro acesso.

Banco de dados

PostgreSQL com volume persistente configurado no Dokploy, garantindo que os dados sobrevivam a reinicializações do container.

CREATE TABLE IF NOT EXISTS tasks (
  id SERIAL PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  done BOOLEAN DEFAULT false,
  created_at TIMESTAMP DEFAULT NOW()
);

Disponibilidade pública

Como o servidor é uma VM local (sem IP público), a exposição à internet foi feita via Cloudflare Tunnel (modo Quick Tunnel), conforme previsto no edital para ambientes sem IP fixo.

cloudflared tunnel --url http://localhost:8080   # frontend
cloudflared tunnel --url http://localhost:8081   # página descritiva

08

Registro de incidentes & resolução

Conforme previsto no edital, documentamos abaixo as principais dificuldades técnicas enfrentadas e como foram resolvidas — parte do processo real de gestão de infraestrutura.

VM inacessível via IP — modo de rede NAT
Adaptador de rede alterado de NAT para Bridge, permitindo acesso direto pela rede local.
Disco saturado (100%) derrubando Dokploy e Postgres interno
Disco virtual redimensionado de 20GB para 40GB via VBoxManage; partição e LVM expandidos com growpart, pvresize, lvextend e resize2fs.
Imagem postgres:18 causando loop de restart (exit code 1)
Substituída pela imagem estável postgres:16-alpine.
Backend não conectava ao banco (ECONNREFUSED em localhost:5432)
Variável DB_HOST corrigida para apontar ao hostname interno do serviço de banco no Swarm, em vez de localhost.
Portas publicadas em modo Ingress não respondiam no host
Alterado o Published Port Mode de Ingress para Host, adequado a um Swarm de nó único.
